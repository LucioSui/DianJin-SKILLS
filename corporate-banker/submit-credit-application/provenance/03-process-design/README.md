# 03 - 流程设计

## 输入

- 原SKILL.md(5步流程)
- 监管要求映射表(来自01-regulatory-review)
- 专家经验(来自02-expert-interview)

## 过程

### 流程重构决策

| 决策项 | 原设计 | 新设计 | 依据 |
|--------|--------|--------|------|
| 步骤数量 | 5步 | 6步(增加步骤0) | 先读后写机制,强制数据验证 |
| 门控机制 | 无明确分支 | 每步标注✅/❌/⚠️分支 | 模式B-步骤门控型交互模式 |
| 确认机制 | 步骤5为explicit | 步骤1存在冲突时为approve | 高风险Skill需人工确认门 |
| 防偷懒指令 | 无 | 步骤0/2/5增加强制指令 | 对抗模型偷懒 |
| 降级策略 | 4种场景 | 4种场景(保留) | 覆盖系统不可用、数据缺失 |

### 门控路径设计

**步骤1(前置状态核查)**:
- ✅ 无活跃案件 → 进入授信到期检查
- ❌ 存在活跃案件 → **立即停止**,提示冲突
- ⚠️ 查询超时 → 标注"未核查",用户手动确认后方可继续

**步骤3(完备性评估)**:
- ✅ "可以提交"或"建议补充后提交" → 进入步骤4
- ❌ "暂缓提交" → **触发红线R1**,立即停止
- ⚠️ 材料存疑 → 标注存疑项,用户决定

**步骤5(确认提交)**:
- ✅ 用户明确说"提交"或"确认" → 发起案件创建
- ❌ 用户说"先不提交" → **终止流程**
- ⚠️ 模糊表述("好的") → **不得判定为确认**,再次询问

### 输出格式结构化

设计6章节表格输出,标注下游可解析字段,确保 credit-approval-decision Skill 可可靠提取。

## 输出

- 门控路径定义(3个关键门控点)
- 结构化输出格式(6章节表格)
