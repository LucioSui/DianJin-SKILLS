# 步骤3:信号分析与报告设计

## 输入
- 核心业务规则提取清单(来自[步骤1](../01-framework-design/README.md))
- 交互模式判定报告:模式A - 报告生成型

## 过程
1. 设计红旗/黄旗信号分级机制:
   - 🚩红旗(重大风险):失信被执行人、裁判文书涉案金额>注册资本30%、近2年行政处罚≥3次、负面舆情(6个月内)
   - ⚠️黄旗(潜在风险):实缴比例<30%、经营异常未解除、法人个人诉讼>50万、授信占用率>90%、征信查询>6次/6个月、对外担保>净资产50%
   - ✅正面信号:高新技术企业、专精特新等荣誉资质
2. 设计一票否决条件V1-V6:
   - V1:企业或实控人被列入失信被执行人名单,且未解除
   - V2:征信系统存在未结清逾期贷款(M3及以上,6个月内)
   - V3:企业被纳入经营异常名录超过1年,且无任何解除记录
   - V4:近2年内有涉嫌金融诈骗、洗钱相关行政处罚或刑事立案记录
   - V5:企业所属行业属于监管明令限制/淘汰类
   - V6:实控人近期出逃、失联,或有相关负面舆情经多源交叉验证属实
3. 设计访前一页纸输出模板:
   - 报告基本信息(报告编号/生成时间/客户名称/走访目的)
   - 企业概况(全称/统一信用代码/法人/实控人/注册资本/经营范围/股权结构)
   - 经营状况(主营业务/经营动态/行内数据摘要/资质荣誉)
   - 风险信号评级(红旗/黄旗/正面信号,逐条列出并注明数据来源)
   - 一票否决检查(V1-V6逐一比对结果)
   - 走访重点建议(必须核实的事项/建议了解的问题/建议收集的材料)
   - 数据来源(五大数据维度状态/来源工具/渠道/备注)
4. 设计先读后写机制(步骤0):
   - 强制运行validate_pre_visit.py验证输入数据
   - 验证客户名称/ID非空,走访目的明确
   - 验证通过后才进入数据收集步骤
