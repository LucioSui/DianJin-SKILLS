# 步骤4:Skill合成与迭代

## 输入
- 7步Workflow设计(来自[步骤3](../03-signal-analysis/README.md))
- 核心业务规则保护清单(8条规则,来自步骤1)
- METHODOLOGY.md金融级标准

## 过程
1. 重构YAML Frontmatter:
   - name:pre-visit-credit-analysis(保留)
   - description:增加触发词+反触发条件(功能+触发+反触发)
   - 新增字段:target_role/business_domain/risk_level/version(2.0.0)/status(draft)/data_sources/upstream_skills/downstream_skills
2. 增加Target Role章节:角色/场景/输出用途/决策层级/执行频率
3. 增加Data Sources章节:5项必需数据+4种降级策略
4. 重构Workflow:5步数据收集→7步(增加步骤0先读后写+步骤6一票否决自检),增加防偷懒指令
5. 强化Constraints:5条→7条(增加禁止跳过步骤/禁止越权建议)
6. 增加Audit Trail:JSON格式审计日志模板
7. 增加Gotchas:4条踩坑记录(股权穿透/舆情搜索/征信授权/走访建议)
8. 增加Examples:2个端到端示例(新客户首笔/存量续贷)
9. 增加Output Format:结构化表格(7章节)+下游兼容性说明+免责声明
10. 增加Out of Scope:5项非功能范围界定
11. 核心业务逻辑保护:8条规则全部保留,未篡改

## 核心业务逻辑保护清单

改造过程中严格保留以下8条核心业务规则,未做任何篡改:

1. **股权穿透至自然人**:至少3层穿透,不得停留在中间层法人
2. **实缴比例阈值**:实缴比例<30%标注黄旗
3. **失信被执行人触发V1**:直接一票否决
4. **行政处罚阈值**:近2年≥3次标注红旗
5. **授信占用率阈值**:>90%标注黄旗
6. **征信查询阈值**:>6次/6个月标注黄旗
7. **对外担保阈值**:>净资产50%标注黄旗
8. **一票否决V1-V6优先检查**:触发即红色标注,不得淡化
